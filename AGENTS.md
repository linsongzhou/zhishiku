# 个人知识库维护规则

## 目标

将原始资料逐步编译为可追溯、可关联、可复用的 Markdown 知识网络；原始资料是证据，`wiki/` 是整理后的知识层。

## 目录职责

- `raw/articles/`：长文章、网页、公众号文章的 Markdown 原文与本地资源。
- `raw/clips/`：灵感、摘抄、金句、临时记录；先快速保存，后续再整理。
- `raw/papers/`：论文、研究报告及其笔记。
- `wiki/sources/`：每一份有价值来源的摘要、要点、出处、适用范围与关联。
- `wiki/concepts/`：概念、方法、主题页；链接多个来源与相关概念。
- `wiki/entities/`：人物、书籍、机构、产品、项目等实体页。
- `wiki/syntheses/`：跨来源的综合理解、决策记录和专题总结。
- `outputs/qa/`：基于知识库的问答与临时分析。
- `outputs/health/`：健康检查报告。

## 维护原则

1. 新资料先保存到 `raw/`，不覆盖或伪造原始出处。
2. 事实性结论必须保留来源链接或本地原文链接；无法确认时标注 `待核验`。
3. 没有来源的信息标注 `待补充`，不得包装为已验证事实。
4. 优先补充既有 wiki 页面；只有出现稳定、可复用的新主题或实体时新建页面。
5. 每次整理资料后更新 `wiki/index.md` 与 `wiki/log.md`。
6. 文件名使用简短、稳定、可读的中文或英文名称；避免仅用日期或随机字符命名。
7. 图片、PDF 等附件放进同名资料目录下的 `assets/`，在 Markdown 中使用相对路径引用。

## 文稿格式

知识库中的笔记、文章、综合报告和问答默认参照 GB/T 9704—2012 编排。该标准的纸张、版心、字体、字号、行距、页码和印刷装订要求不直接适用于 Markdown；转换为 Word、PDF 或正式公文时再按原标准恢复。

1. 标题应准确概括内容，一般不使用标点符号；层级应简明、稳定，不得跳级。
2. 正文层次序号依次使用“一、”“（一）”“1.”“（1）”；知识库的 Markdown 标题可在序号前保留 `#` 标记。
3. 数字、年份、日期、时间、计量单位和标点符号应采用国家标准规定的规范写法；标准编号中的年代号使用长横线“—”，如“GB/T 9704—2012”。
4. 引用原文、标准条文和网络资料时，应标明来源、版本、条款或页码；推断、经验判断和未核实信息应分别标注“推断”“经验判断”或“待核验”。
5. 同一文稿中的术语、单位、符号、编号和日期格式应保持一致。

## 最小闭环

一条有价值资料的处理顺序：

`raw → wiki/sources → wiki/concepts 或 entities → wiki/index → wiki/log`

## 健康检查

定期检查：未整理的 raw 资料、缺来源的 wiki 页面、孤儿页面、失效链接、重复主题和长期待核验项。报告保存到 `outputs/health/`。
